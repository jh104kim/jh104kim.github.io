## 코드 리뷰 태스크

**대상 결정:**
- `$ARGUMENTS`가 비어 있거나 `staged`이면 → `git diff --staged` 로 staged 변경 전체를 리뷰
- `$ARGUMENTS`가 파일 경로면 → 해당 파일을 직접 읽어서 리뷰
- `$ARGUMENTS`가 `HEAD` 또는 커밋 해시면 → `git diff HEAD~1 HEAD` 또는 지정 범위를 리뷰

**수행 절차:**
1. 대상 코드/diff를 확인한다.
2. 아래 3가지 관점으로 분석한다.
3. 결과를 **한국어**로 심각도 이모지와 함께 보고한다.

---

## 리뷰 관점

### 🔴 버그 / 런타임 오류
- Null/undefined 역참조, 배열 범위 초과, 타입 불일치
- 예외 미처리, 무한루프, 비동기 경쟁 조건(race condition)
- 논리 오류, 잘못된 조건식

### 🟡 보안 취약점
- 입력값 미검증(XSS, SQL Injection, Command Injection)
- 민감 정보 하드코딩(API 키, 비밀번호, 토큰)
- 불필요한 권한 확대, 인증/인가 누락
- 안전하지 않은 의존성 사용

### 🟢 개선 제안
- 중복 코드, 불필요한 복잡도
- 성능 개선 여지 (N+1, 불필요한 재계산)
- 가독성·유지보수성 향상 포인트
- Next.js / TypeScript / React 모범 사례 미준수
